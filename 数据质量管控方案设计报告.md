## 二、项目数据质量管控背景

农产品价格采集监控平台需要从公开数据文件、接口或网页中获取农产品价格数据，并结合气象辅助数据完成价格趋势分析、地区对比、气象影响分析和价格预测。由于农产品价格受市场、地区、季节、天气和采集来源影响较大，原始数据容易出现字段缺失、单位不统一、重复采集、异常价格、日期格式不一致、地区名称不规范等问题。

如果数据质量没有得到有效控制，后续会直接影响以下环节：

1. 数据入库时可能出现字段缺失、类型错误或记录数不一致。
2. 价格趋势图和地区对比图可能因重复记录或异常值产生偏差。
3. 价格与气象数据按地区和日期关联时可能出现匹配失败。
4. 机器学习模型可能因为异常价格或缺失特征导致预测误差增大。
5. 前端展示指标可能与后台数据、测试报告和项目文档口径不一致。

因此，本项目需要设计一套面向实训项目的轻量化数据质量管控方案，在本地可运行实现的基础上，保留向 Kafka、HDFS、Spark、MySQL、Redis 等完整大数据平台扩展的能力。

## 三、数据质量管控目标

### 3.1 总体目标

本项目的数据质量管控目标是：在农产品价格数据从采集到展示的完整链路中，对数据来源、字段结构、数据格式、缺失值、重复值、异常值、关联结果、入库一致性和分析结果进行持续检查，保证平台所使用的数据真实可用、结构清晰、处理过程可复现、质量结果可验证。

### 3.2 具体目标

| 目标类别 | 管控目标 | 本组对应措施 |
| --- | --- | --- |
| 完整性 | 核心字段不能缺失，数据规模满足实训要求 | 检查产品、地区、日期、价格、单位、气象字段，价格数据不少于 1000 条 |
| 准确性 | 日期、价格、温度、降雨量等字段格式正确且数值合理 | 使用 pandas 转换日期和数值字段，采用 IQR 方法修正异常值 |
| 一致性 | 字段命名、日期格式、价格单位和地区口径保持统一 | 统一 date 为 YYYY-MM-DD，统一 unit 为元/公斤 |
| 唯一性 | 避免重复记录影响统计结果 | 清洗阶段删除完全重复记录，质量检查统计重复比例 |
| 可关联性 | 价格数据与气象数据能够按 region 和 date 关联 | 生成 merged_data.csv，关联后记录数与价格数据一致 |
| 可追溯性 | 原始数据、清洗数据、报告和入库结果可以回溯 | 保留 data/raw、data/processed、cleaning_report.md 和测试记录 |
| 可验证性 | 数据质量规则能够通过自动化测试验证 | 使用 pytest 和 Hypothesis 验证清洗规则、质量阈值和入库一致性 |

## 五、数据质量评价维度

结合农产品价格监控业务和实训要求，本项目从以下维度评价数据质量：

| 质量维度 | 含义 | 检查内容 | 典型问题 |
| --- | --- | --- | --- |
| 完整性 | 数据是否覆盖业务所需字段和记录范围 | 字段是否存在、必填字段是否为空、记录数是否达标 | 缺少 average_price、region、date 等核心字段 |
| 准确性 | 数据值是否符合真实业务含义和数值范围 | 价格是否为正数，温度、降雨、湿度是否合理 | 平均价格为负数，湿度大于 100% |
| 一致性 | 同类数据在不同来源和环节中是否口径统一 | 日期格式、价格单位、地区名称、字段名称 | 元/斤和元/公斤混用，日期格式混乱 |
| 唯一性 | 同一业务记录是否被重复保存 | 完全重复记录、同产品同地区同日期重复记录 | 重复采集导致价格均值偏高或偏低 |
| 及时性 | 数据是否覆盖需要分析的时间范围 | 日期范围、采集时间、更新周期 | 缺少最近日期数据，无法展示趋势 |
| 可关联性 | 不同主题数据是否可以正确连接 | price_data 与 weather_data 的 region、date 匹配 | 地区名不一致导致合并后记录丢失 |
| 可追溯性 | 数据从来源到结果是否有记录可查 | 原始文件、清洗文件、入库日志、测试记录 | 统计结果无法追溯到源数据 |
| 可用性 | 数据是否能支持查询、分析、预测和展示 | 入库成功率、接口返回、模型训练数据量 | 数据可保存但无法用于前端图表 |

## 六、数据质量标准与阈值设计

### 6.1 采集层质量标准

| 检查项 | 质量标准 | 管控方式 |
| --- | --- | --- |
| 价格数据规模 | 清洗后价格数据不少于 1000 条 | 在采集测试和入库测试中统计记录数 |
| 农产品数量 | 至少覆盖 3 种农产品 | 检查 product_name 去重数量 |
| 地区数量 | 至少覆盖 2 个地区，本组实际覆盖 3 个地区 | 检查 region 去重数量 |
| 时间范围 | 至少覆盖 6 个月，本组覆盖 2023-11-03 至 2024-06-30 | 检查 date 最小值和最大值 |
| 数据来源 | 原始数据和来源说明需要保留 | 维护 data/raw 和数据来源说明文档 |

### 6.2 字段完整性标准

价格数据必填字段如下：

| 字段 | 质量要求 |
| --- | --- |
| product_name | 不为空，用于产品维度查询和趋势分析 |
| product_category | 不为空，用于产品类别统计 |
| market_name | 不为空，用于市场来源识别 |
| region | 不为空，用于地区对比和气象关联 |
| date | 不为空，统一为 YYYY-MM-DD 格式 |
| average_price | 不为空，必须为可计算数值 |
| unit | 统一为元/公斤 |

气象数据必填字段如下：

| 字段 | 质量要求 |
| --- | --- |
| region | 不为空，并与价格数据地区口径一致 |
| date | 不为空，并与价格数据日期口径一致 |
| average_temperature | 可为空但应尽量完整，用于气象影响分析 |
| rainfall | 可为空但应为非负数，用于气象影响分析 |
| humidity | 可为空但应处于合理范围，用于气象影响分析 |

### 6.3 缺失值质量标准

| 规则编号 | 规则内容 | 当前配置 |
| --- | --- | --- |
| Q-MISS-01 | 单个字段缺失比例小于 50% 时保留字段 | missing_threshold = 0.50 |
| Q-MISS-02 | 数值字段缺失比例小于 50% 时使用均值填充 | pandas 均值填充 |
| Q-MISS-03 | 分类字段缺失比例小于 50% 时使用众数填充 | pandas 众数填充 |
| Q-MISS-04 | 字段缺失比例大于等于 50% 时删除字段 | cleaning 阶段删除 |
| Q-MISS-05 | 清洗后整体缺失比例不得超过 20% | quality_max_missing = 0.20 |

### 6.4 重复值质量标准

| 规则编号 | 规则内容 | 当前配置 |
| --- | --- | --- |
| Q-DUP-01 | 清洗阶段删除完全重复的记录 | drop_duplicates |
| Q-DUP-02 | 清洗后重复记录比例不得超过 10% | quality_max_duplicate = 0.10 |
| Q-DUP-03 | 入库后记录数应与清洗后 CSV 数据行数一致 | verify_import 对比记录数 |

### 6.5 异常值质量标准

| 规则编号 | 规则内容 | 当前配置 |
| --- | --- | --- |
| Q-OUT-01 | 价格字段 highest_price、lowest_price、average_price 参与异常检测 | PRICE_NUMERIC_COLUMNS |
| Q-OUT-02 | 气象字段 temperature、rainfall、humidity、sunshine_duration 等参与异常检测 | WEATHER_NUMERIC_COLUMNS |
| Q-OUT-03 | 使用 IQR 方法识别异常值 | IQR = Q3 - Q1 |
| Q-OUT-04 | 下界为 Q1 - 1.5 × IQR，上界为 Q3 + 1.5 × IQR | iqr_multiplier = 1.5 |
| Q-OUT-05 | 超出边界的异常值修正为边界值 | clip 到上下界 |

### 6.6 格式一致性标准

| 检查项 | 标准格式 | 管控方式 |
| --- | --- | --- |
| 日期字段 | YYYY-MM-DD | 使用 pandas.to_datetime 转换 |
| 价格单位 | 元/公斤 | 将元/斤乘以 2，元/吨除以 1000 |
| 价格字段 | 数值型，保留合理小数 | 清洗和入库前转换为数值 |
| 地区字段 | 山东、河南、四川等统一地区名称 | 清洗后用于 groupby 和关联 |
| 字段名称 | 英文字段名，与数据库表结构一致 | DBImporter 导入时筛选合法字段 |

## 七、数据质量管控总体架构

### 7.1 完整实训平台管控架构

完整农产品价格采集监控平台可采用如下质量管控架构：

```text
数据源/API/网页
        ↓
采集质量校验：字段存在、来源记录、采集时间、响应状态
        ↓
Kafka 消息缓冲：记录原始消息、异常消息和重试状态
        ↓
Spark/Python 清洗：缺失值处理、去重、异常值修正、格式统一
        ↓
HDFS/CSV 归档：保存原始数据、清洗数据、失败数据和历史版本
        ↓
MySQL 入库校验：表结构校验、字段类型校验、记录数一致性校验
        ↓
Redis/API/前端：热点指标校验、图表数据校验、接口返回结构校验
        ↓
质量报告与告警：输出清洗报告、入库日志、质量检查日志和测试报告
```

### 7.2 本组当前落地管控架构

结合当前项目规模和本地可运行要求，本组采用以下轻量化管控方案：

| 管控环节 | 当前实现 | 质量作用 |
| --- | --- | --- |
| 原始数据保留 | data/raw/price_data.csv、data/raw/weather_data.csv | 保证数据来源可回溯 |
| 数据清洗 | backend/data_cleaner.py | 处理缺失值、异常值、重复值和格式统一 |
| 清洗结果输出 | data/processed/processed_price.csv、processed_weather.csv | 作为入库和分析的标准数据 |
| 合并数据验证 | data/processed/merged_data.csv | 验证价格数据与气象数据可关联 |
| 清洗报告 | data/processed/cleaning_report.md | 记录清洗前后记录数、缺失比例、重复比例和异常修正数量 |
| 入库校验 | backend/db_importer.py | 校验必要字段、批量导入和记录数一致性 |
| 自动化测试 | tests/properties/test_quality_check.py、test_data_cleaning.py、test_db_import.py | 验证清洗规则、质量阈值和入库一致性 |
| 展示校验 | 前端 Vue + ECharts 页面 | 通过图表和指标检查数据是否可用 |

## 八、全流程质量管控方案

### 8.1 数据采集阶段

采集阶段的重点是保证数据来源清楚、字段完整、记录规模达标。

| 管控内容 | 管控规则 | 异常处理 |
| --- | --- | --- |
| 数据来源 | 记录来源名称、采集方式、字段说明和时间范围 | 来源不可用时使用已归档 CSV 作为备份 |
| 字段结构 | 检查是否包含产品、地区、日期、价格、单位等字段 | 缺少核心字段时拒绝进入清洗流程 |
| 数据规模 | 价格数据不少于 1000 条，产品不少于 3 种，地区不少于 2 个 | 数据不足时补充采集或扩大时间范围 |
| 原始归档 | 原始数据先保存到 data/raw | 禁止直接覆盖清洗后数据 |

完整平台中，爬虫或 API 采集任务应增加任务状态、响应码、采集耗时和失败原因记录，并将异常消息写入 Kafka 的异常 Topic 或 quality_check_log 表。

### 8.2 数据清洗阶段

清洗阶段是本项目数据质量管控的核心。backend/data_cleaner.py 中的 DataCleaner 模块负责执行以下规则：

1. 记录清洗前数据行数。
2. 检查每个字段缺失比例。
3. 对缺失比例小于 50% 的数值字段使用均值填充。
4. 对缺失比例小于 50% 的分类字段使用众数填充。
5. 删除缺失比例大于等于 50% 的字段。
6. 对价格和气象数值字段使用 IQR 方法修正异常值。
7. 删除完全重复记录。
8. 将日期字段统一为 YYYY-MM-DD。
9. 将价格单位统一为元/公斤。
10. 输出清洗报告，记录清洗结果。

清洗完成后，数据应满足以下要求：

| 检查项 | 达标标准 |
| --- | --- |
| 清洗后缺失比例 | 不超过 20% |
| 清洗后重复比例 | 不超过 10% |
| 日期格式 | 全部可解析为 YYYY-MM-DD |
| 价格单位 | 全部统一为元/公斤 |
| 关键字段 | product_name、region、date、average_price 等字段可用 |

### 8.3 数据合并阶段

价格数据和气象数据需要按 region 和 date 进行关联，合并后的数据用于气象影响分析和价格预测。

| 管控内容 | 管控规则 | 当前结果 |
| --- | --- | --- |
| 关联字段 | price_data.region = weather_data.region，price_data.date = weather_data.date | 已按地区和日期关联 |
| 合并方式 | 使用内连接，保证每条记录同时包含价格和气象信息 | merged_data.csv 已生成 |
| 合并记录数 | 合并后记录数应与可关联价格记录数一致 | 当前为 2169 条 |
| 合并缺失值 | 合并后缺失比例不得超过 20% | 当前为 0.0000 |

如果后续扩展更多地区或数据源，需要增加地区名称映射表，避免“山东省”和“山东”等口径不同导致数据无法关联。

### 8.4 数据入库阶段

入库阶段的重点是保证清洗数据能够按照标准表结构稳定写入数据库，并且入库结果与输入文件一致。

| 管控内容 | 管控规则 | 当前实现 |
| --- | --- | --- |
| 表结构校验 | price_data 和 weather_data 表字段覆盖核心数据 | backend/schema.sql |
| 必填字段校验 | 缺少必要字段时拒绝导入 | DBImporter 字段检查 |
| 字段筛选 | 只导入数据库表中定义的字段 | PRICE_COLUMNS、WEATHER_COLUMNS |
| 批量导入 | 使用 pandas to_sql，批大小为 1000 行 | db_importer.py |
| 记录数验证 | 入库后 count(*) 与 CSV 行数一致 | verify_import |
| 降级方案 | MySQL 不可用时使用 SQLite 完成测试 | SQLite 临时数据库 |

当前入库测试结果显示：价格数据和气象数据能够成功导入测试数据库，数据库记录数与 CSV 输入行数保持一致，基础查询和价格气象关联查询均通过。

### 8.5 数据分析阶段

数据分析阶段需要保证统计结果不受脏数据影响。

| 分析任务 | 质量要求 |
| --- | --- |
| 平均价格统计 | average_price 必须为有效数值，不能包含异常极端值 |
| 价格趋势分析 | 同一产品、地区、日期维度数据应连续可用 |
| 地区价格对比 | region 字段口径统一，避免地区重复分组 |
| 气象影响分析 | 气象字段应经过异常值修正，价格与气象数据可关联 |
| 异常波动识别 | 先保证基础价格数据准确，再计算涨跌幅 |

分析结果应与后端 API、前端图表和测试记录保持一致。例如，本组当前按产品统计时，番茄、玉米、苹果各有 723 条价格记录；按地区统计时，山东、河南、四川各有 723 条价格记录。

### 8.6 模型预测阶段

模型预测依赖历史价格和气象特征，因此需要在建模前进行特征数据质量检查。

| 管控内容 | 质量要求 |
| --- | --- |
| 训练数据量 | 模型训练记录数不少于配置中的 min_training_records |
| 特征完整性 | average_price、region、date、temperature、rainfall、humidity 等关键特征可用 |
| 目标变量 | average_price 不能缺失，且应为合理价格数值 |
| 时间顺序 | 训练和预测应按日期排序，避免时间序列错乱 |
| 模型评估 | 输出 MAE、MSE、RMSE 等指标，检查预测误差是否可接受 |

如果模型预测误差明显偏大，应优先检查数据质量问题，包括异常价格是否未处理、气象数据是否缺失、训练样本是否不足、同一产品不同地区数据是否混合不当。

### 8.7 可视化展示阶段

前端展示阶段需要保证图表数据与后端接口返回一致，避免用户看到错误指标。

| 管控内容 | 质量要求 |
| --- | --- |
| 首页指标 | 数据总量、产品数、地区数、平均价格应来自同一数据口径 |
| 趋势图 | 日期按时间顺序展示，价格单位明确 |
| 地区对比图 | 地区名称不重复，均价计算方式一致 |
| 气象影响图 | 气象字段和价格字段来自同一 region + date 关联数据 |
| 预测图 | 真实值和预测值使用相同产品、地区和时间范围 |

前端页面如果出现图表为空、坐标轴异常或数据突变，应回到后端接口和 processed 数据文件进行排查。

## 十、自动化测试与验收方案

### 10.1 数据清洗规则测试

测试文件 tests/properties/test_data_cleaning.py 用于验证数据清洗规则正确性，主要覆盖：

1. 缺失比例低于 50% 的数值字段填充后无空值。
2. 缺失比例大于等于 50% 的字段被删除。
3. 价格和气象数值字段中的异常值被修正到 IQR 边界范围内。
4. 完全重复记录被删除。
5. 清洗后保留字段不应存在未处理空值。

### 10.2 数据质量阈值测试

测试文件 tests/properties/test_quality_check.py 用于验证质量阈值判定规则：

| 条件 | 预期结果 |
| --- | --- |
| 缺失比例 <= 20% 且重复比例 <= 10% | 质量达标 |
| 缺失比例 > 20% | 质量不达标 |
| 重复比例 > 10% | 质量不达标 |
| 缺失比例和重复比例同时超标 | 质量不达标 |

### 10.3 数据入库一致性测试

数据入库一致性测试用于验证：

1. processed_price.csv 可以导入 price_data 表。
2. processed_weather.csv 可以导入 weather_data 表。
3. 入库后记录数与 CSV 数据行数一致。
4. SQLite 降级测试库可以在 MySQL 不可用时保证测试流程继续运行。

已有入库测试输出摘要如下：

```text

```

## 十一、质量问题处理流程

当数据质量检查发现问题时，按以下流程处理：

```text
发现质量问题
        ↓
定位问题类型：缺失、重复、异常、格式错误、关联失败、入库失败
        ↓
查看原始数据和清洗报告
        ↓
判断问题来源：采集源、字段转换、清洗规则、数据库结构或接口展示
        ↓
制定修复措施：重新采集、补充字段、修正规则、调整阈值或修复表结构
        ↓
重新执行清洗和入库
        ↓
运行自动化测试
        ↓
更新质量报告和测试记录
```

常见质量问题及处理措施如下：

| 问题类型 | 表现 | 处理措施 |
| --- | --- | --- |
| 字段缺失 | CSV 中缺少 product_name、date、average_price 等字段 | 拒绝入库，返回采集阶段补充字段 |
| 缺失值过多 | 某字段缺失比例大于等于 50% | 删除字段或重新采集数据 |
| 重复记录 | 重复比例超过 10% | 清洗阶段去重，并检查采集任务是否重复执行 |
| 异常价格 | 价格为负数或明显偏离正常范围 | 使用 IQR 修正，必要时人工核查来源 |
| 单位不统一 | 元/斤、元/吨、元/公斤混用 | 统一换算为元/公斤 |
| 日期格式错误 | 日期无法解析或格式混乱 | 统一转换为 YYYY-MM-DD，无法解析的记录单独记录 |
| 关联失败 | 价格与气象按 region、date 合并后记录减少 | 检查地区名称和日期范围是否一致 |
| 入库失败 | 字段类型不匹配或必要字段缺失 | 校验 schema.sql 和 DBImporter 字段配置 |

## 十三、扩展质量管控设计

当前项目以本地 CSV、pandas、MySQL/SQLite 和 pytest 为主。若后续扩展为完整大数据平台，可增加以下质量管控能力：

| 扩展方向 | 设计内容 | 作用 |
| --- | --- | --- |
| Kafka 质量缓冲 | 设置 raw_price_topic、invalid_price_topic、quality_alert_topic | 区分正常数据、异常数据和质量告警 |
| HDFS 分区归档 | 按 source、date、status 分区保存原始和清洗数据 | 便于追溯历史版本和失败数据 |
| Spark 批量质量校验 | 使用 Spark SQL 统计空值、重复值、异常值和分布变化 | 支持更大规模数据质量检查 |
| MySQL 质量日志表 | 增加 quality_check_log、import_log、crawler_task 表 | 保存每次采集、清洗和入库的质量结果 |
| Redis 质量指标缓存 | 缓存最近一次质量评分、异常记录数和首页指标 | 提高监控页面响应速度 |
| 质量告警机制 | 当缺失率、重复率、异常率超过阈值时告警 | 提醒成员及时处理质量问题 |
| 数据血缘记录 | 记录原始文件、清洗文件、入库批次和前端指标之间的关系 | 支持问题追踪和答辩说明 |

建议扩展的 quality_check_log 表结构如下：

| 字段名 | 类型 | 说明 |
| --- | --- | --- |

| total_records | INT | 总记录数 |
| missing_ratio | DECIMAL(6,4) | 缺失值比例 |
| duplicate_ratio | DECIMAL(6,4) | 重复记录比例 |
| outlier_count | INT | 异常值数量 |
| is_acceptable | BOOLEAN | 是否达标 |
| issue_detail | TEXT | 问题说明 |

## 十四、风险分析与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 数据源结构变化 | 采集字段缺失，清洗脚本失败 | 增加字段存在性检查，保留 CSV 备份数据 |
| 数据规模不足 | 无法满足实训数据量要求 | 扩展采集时间范围、产品种类和地区数量 |
| 异常值处理过度 | 真实价格波动被误修正 | 对重要异常价格增加人工核查，保留原始数据 |
| 地区名称不统一 | 价格和气象数据无法正确关联 | 建立地区名称标准化映射表 |
| MySQL 环境不可用 | 入库和演示受阻 | 使用 SQLite 降级方案保证测试可运行 |
| 文档与代码不一致 | 验收时难以解释 | 以 cleaning_report、测试记录和代码配置为准定期更新文档 |
| 前端图表数据异常 | 展示效果不可信 | 从前端接口反查数据库和 processed 文件 |
