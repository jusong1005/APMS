# 农产品价格采集监控平台后端编写文档

## 1. 文档目的

本文档用于说明本项目后端的编写方案、模块划分、接口设计、数据流转方式和开发步骤。当前前端页面已经完成整体壳子和主要业务页面，但后端服务还没有真正落地，因此本文档面向后续 **Spring Boot 3 + JDK 17** 的后端实现。

本项目当前的数据链路如下：

```text
Python 采集 -> Kafka 原始 Topic -> Scala/Spark 清洗 -> MongoDB / Redis / HDFS
                        \-> Spark 历史分析 -> HDFS 结果
```

后端的职责不是重复清洗和分析，而是：

1. 为前端提供统一 API。
2. 管理用户注册、登录认证、采集任务、预警规则、用户权限和系统配置。
3. 查询 MongoDB、Redis、HDFS 中的结果并返回给前端。
4. 必要时触发批量任务、导出任务和分析任务。
5. 记录审计日志和操作日志。

## 2. 技术选型

### 2.1 推荐技术栈

| 模块 | 技术 | 说明 |
| --- | --- | --- |
| 后端框架 | Spring Boot 3.x | 作为主后端服务 |
| Java 版本 | JDK 17 | 与当前服务器环境一致 |
| 构建工具 | Maven 3.9.x | 便于打包和依赖管理 |
| 数据库 | MongoDB 7.x | 存储业务数据、任务配置、日志、用户信息 |
| 缓存 | Redis 7.x | 存储实时批次指标、预警和热点数据 |
| 分析结果 | HDFS | 存储 Spark 历史分析输出 |
| 消息队列 | Kafka 3.7.x | 实时采集与清洗链路缓冲 |
| 任务调度 | Spring `@Scheduled` | 单节点场景优先使用 |
| 安全认证 | Spring Security + JWT | 实现注册、登录、令牌续期和权限控制 |
| 密码加密 | BCrypt | 保存密码摘要，不保存明文密码 |
| 文件上传 | 本地目录 / 对象存储 | 用于头像、导出文件、附件等 |

### 2.2 设计原则

1. 先满足单节点部署和课程演示。
2. 后端只做业务编排，不重复实现 Spark 清洗逻辑。
3. 前端所有页面尽量通过统一 REST API 获取数据。
4. 读多写少的数据优先缓存到 Redis。
5. 批量分析结果优先写入 HDFS，再由后端读取和分发。

## 3. 后端职责划分

后端建议按业务域拆分为以下模块：

1. **总览监控模块**：提供首页指标、实时趋势、预警摘要。
2. **采集任务管理模块**：管理采集任务的启动、停止、状态和日志。
3. **数据分析模块**：提供地区分析、品类分析、季节性分析、波动分析等查询接口。
4. **趋势预测模块**：提供预测结果、影响因子和预警阈值判断。
5. **预警中心模块**：管理预警规则、预警记录和预警处理状态。
6. **认证登录模块**：提供注册、登录、退出、刷新令牌和当前用户信息。
7. **用户权限模块**：管理用户、角色、机构和权限。
8. **系统配置模块**：管理全局配置、质量阈值、数据库状态和审计日志。
9. **个人中心模块**：维护个人资料、密码、头像和通知偏好。
10. **文件与导出模块**：导出看板快照、分析结果和日志文件。

### 5.6 注册与登录接口

注册和登录是后端必须优先实现的基础能力。前端登录成功后保存 `accessToken`，后续所有需要权限的请求都在请求头中携带：

```text
Authorization: Bearer <accessToken>
```

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| POST | `/api/auth/register` | 用户注册，提交账号、密码、姓名、手机号、邮箱、所属机构 |
| POST | `/api/auth/login` | 用户登录，返回 accessToken、refreshToken 和用户信息 |
| POST | `/api/auth/logout` | 用户退出登录，使当前 refreshToken 失效 |
| POST | `/api/auth/refresh-token` | 使用 refreshToken 换取新的 accessToken |
| GET | `/api/auth/me` | 获取当前登录用户信息、角色和权限 |
| POST | `/api/auth/send-code` | 发送注册或找回密码验证码，可选实现 |
| POST | `/api/auth/forgot-password` | 发起找回密码流程，可选实现 |
| POST | `/api/auth/reset-password` | 使用验证码重置密码，可选实现 |
| PUT | `/api/auth/password` | 登录状态下修改当前用户密码 |

注册流程建议：

1. 校验账号、手机号、邮箱是否重复。
2. 校验密码强度。
3. 使用 BCrypt 生成密码摘要。
4. 默认角色设置为普通用户，或进入待审核状态。
5. 写入 `user_accounts` 集合。
6. 记录 `audit_logs` 审计日志。

登录流程建议：

1. 根据账号、手机号或邮箱查询用户。
2. 判断账号是否启用。
3. 使用 BCrypt 校验密码。
4. 生成 accessToken 和 refreshToken。
5. refreshToken 摘要写入 `login_sessions` 集合或 Redis。
6. 更新最后登录时间、登录 IP 和登录设备。
7. 返回用户基本信息、角色和权限列表。

权限校验建议：

1. 普通用户只能查看监控大盘、价格分析和预警结果。
2. 数据分析员可以手动运行流水线、导出分析结果和处理预警。
3. 管理员可以管理用户、系统配置、预警规则和任务调度。

## 6. 数据设计建议

建议后端主要使用 MongoDB 存储业务配置和操作数据，集合可设计为：

| 集合 | 用途 |

## 8. 开发步骤建议

### 阶段 0：先做注册登录与认证基础

优先实现：

1. 用户注册接口。
2. 用户登录接口。
3. JWT accessToken 生成和校验。
4. refreshToken 续期和退出登录。
5. 当前用户信息 `/api/auth/me`。
6. 基础角色权限：管理员、数据分析员、普通用户。

注册登录完成后，前端才能安全地访问用户权限、系统配置、任务管理等页面。

### 阶段 1：先做只读接口

优先实现以下页面：

1. 监控大盘。
2. 价格数据分析。
3. 价格预警中心。
4. 系统配置中的数据库状态。

这一阶段主要是查询 MongoDB、Redis 和 HDFS，不涉及复杂写操作。

### 阶段 2：补任务管理和用户体系

继续实现：

1. 采集任务管理。
2. 用户权限管理。
3. 个人设置。
4. 审计日志。

这一阶段需要基于阶段 0 的登录态继续补权限判断和任务状态持久化。

### 阶段 3：补预测和导出

最后实现：

1. 趋势预测。
2. 导出快照。
3. 图表导出。
4. 批量任务调度。

## 9. 与现有代码的衔接方式

当前项目里已经有这些现成能力：

1. Python 实时采集生产者，负责把原始数据写入 Kafka。
2. Scala/Spark 实时清洗作业，负责写 MongoDB 和 Redis。
3. Scala/Spark 历史分析作业，负责写 HDFS 历史分析结果。
4. 前端 Vue 页面，负责展示看板、任务、分析、预警、用户、设置和个人中心。

后端 Spring Boot 只需要做一层统一编排：

- MongoDB 负责业务数据和配置数据。
- Redis 负责实时缓存和预警热点。
- HDFS 负责历史分析结果读取。
- Spark 任务通过后端触发或读取结果。
- 前端只调用 Spring Boot API，不直接访问 Kafka、MongoDB 或 HDFS。

## 10. 运行与部署建议

1. 后端端口建议使用 `8080`。
2. 前端通过 Nginx 访问，Nginx 反向代理到 Spring Boot。
3. MongoDB、Redis、Kafka、HDFS 在单节点服务器上以本机或内网方式部署。
4. 后端日志建议写入 `/var/log/agri/`。
5. 登录认证建议从开发初期就接入 Spring Security + JWT。
6. JWT 密钥、过期时间、refreshToken 过期时间不要写死在代码中，应放入 `application.yml` 或环境变量。

## 12. 结论

本项目后端最合适的实现方式是：**Spring Boot 作为业务 API 层，MongoDB 作为主存储，Redis 作为实时缓存，HDFS 作为历史分析结果层，Spark 负责清洗与分析，前端只负责展示和操作入口。**

这套方案既符合当前单节点部署，也方便后续扩展成完整平台。
