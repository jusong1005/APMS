## 三、数据特征分析

### 3.1 实训平台数据特征

实训平台面向农产品价格采集监控场景，数据具有以下特点：

1. 来源多样：数据可能来自农业农村部门网站、批发市场网站、电商平台、第三方价格周报或 API。
2. 更新频繁：价格数据具有时效性，实训要求中提到更新频率可达到 30 分钟级别。
3. 字段结构化明显：核心字段包括农产品名称、规格、价格、单位、采集时间、市场名称、地区、数据来源等。
4. 质量差异较大：不同来源的数据可能存在缺失字段、重复记录、单位不一致、时间格式不统一等问题。
5. 冷热数据并存：近期价格数据需要频繁查询，历史原始数据更适合归档保存。

## 四、总体存储架构设计

### 4.1 实训标准架构

按照农产品价格采集监控平台的实训要求，完整平台可采用“Kafka 消息缓冲 + Spark 清洗处理 + HDFS 历史归档 + MySQL 结构化查询 + Redis 热点缓存”的分层存储架构。

| 层级 | 技术组件 | 存储内容 | 作用 |
| --- | --- | --- | --- |
| 采集缓冲层 | Kafka | 爬虫采集到的价格消息、异常消息 | 解耦采集端和处理端，支持实时数据流转 |
| 原始归档层 | HDFS | 原始爬虫数据、原始 CSV、历史批量文件 | 保存全量历史数据，便于追溯和离线分析 |
| 清洗计算层 | Spark/Spark SQL | 清洗中间结果、统计结果 | 完成去重、缺失值处理、单位转换和批量分析 |
| 结构化存储层 | MySQL | 农产品价格表、市场表、地区表、预测结果表 | 支持后端接口、条件查询和前端展示 |
| 热点缓存层 | Redis | 热门农产品价格、首页指标、预警结果 | 提高高频查询响应速度 |
| 本地降级层 | SQLite/CSV | 本地演示数据、自动化测试数据 | 保证实训演示和测试流程可复现 |

### 4.2 本组落地实现

结合课程周期、本地环境和当前数据规模，本组采用可运行、可验证的轻量化落地方案：

| --- | --- | --- | --- |

## 五、数据库选型设计

### 5.1 MySQL 作为主数据库

MySQL 用于保存清洗后的结构化农产品价格数据、气象数据和后续预测结果。选用原因如下：

1. 农产品价格数据字段清晰，适合使用关系型表保存。
2. 平台需要按产品、地区、日期、市场等条件进行组合查询，MySQL 索引可以提高查询效率。
3. MySQL 能与后端服务直接集成，便于为 Vue 前端提供数据接口。
4. MySQL 8.0 开源免费，部署难度低于完整 Hadoop 集群，适合课程实训落地。
5. 后续可将 MySQL 中的核心结构化数据同步到 Hive 或 Spark SQL 中进行大规模分析。

### 5.2 HDFS 作为历史归档扩展

在完整实训平台中，HDFS 用于保存原始爬虫数据、批量历史数据和清洗中间文件。它适合存储大量、低频访问、需要长期留存的数据。本组当前使用 data/raw 和 data/processed 目录模拟 HDFS 分层归档，后续可按日期和数据源迁移到 HDFS 目录，例如：

```text
/agri_price/raw/yyyyMMdd/
/agri_price/processed/yyyyMMdd/
/agri_price/merged/yyyyMMdd/
```

### 5.3 Kafka 作为实时采集缓冲扩展

在完整平台中，爬虫模块可将采集到的数据写入 Kafka Topic，数据清洗模块作为消费者读取消息并处理。这样可以避免爬虫采集速度和清洗入库速度不一致导致阻塞。本组当前使用 CSV 文件作为批量输入，后续可扩展为：

```text
Crawler/API -> Kafka Topic -> Spark/Python Cleaner -> MySQL/HDFS
```

### 5.4 Redis 作为缓存扩展

Redis 适合缓存首页指标、热门农产品价格、最新预警信息和高频查询结果。本组当前数据规模较小，前端可以直接从后端接口查询数据库；若数据量和访问量增加，可将以下数据放入 Redis：

1. 首页价格概览指标。
2. 热门农产品最近 7 天价格走势。
3. 地区价格排行榜。
4. 价格波动超过阈值的预警记录。

### 5.5 SQLite 降级方案

考虑到课程演示时可能出现 MySQL 服务未启动、账号密码不一致或环境配置失败等问题，本组实现了 SQLite 降级方案。当 MySQL 不可用时，系统可自动使用本地 SQLite 数据库继续完成建表、导入、查询和自动化测试，保证实训验收流程不被环境问题阻断。

## 七、数据入库流程设计

### 7.1 实训平台完整流程

```text
爬虫/API/CSV 数据源
        ↓
Kafka 消息队列或本地原始 CSV
        ↓
Spark/Python 数据清洗：去重、缺失值处理、单位转换、格式校验
        ↓
HDFS 保存原始数据和清洗历史文件
        ↓
MySQL 保存结构化价格数据、气象数据、预警结果和预测结果
        ↓
Redis 缓存热点指标
        ↓
后端 API 和前端可视化展示
```

### 7.2 本组当前入库流程

本组的数据入库由 backend/db_importer.py 中的 DBImporter 模块完成：

1. 读取 data/processed/processed_price.csv 和 data/processed/processed_weather.csv。
2. 根据 backend/schema.sql 创建 price_data 和 weather_data 两张表。
3. 对导入字段进行筛选，只保留数据库表结构中定义的字段。
4. 检查必要字段是否存在，例如价格数据必须包含 product_name、product_category、market_name、region、date、average_price。
5. 使用 pandas 的 to_sql 方法进行批量导入，批大小设置为 1000 行。
6. 导入完成后调用 verify_import 方法，对比数据库表记录数和 CSV 数据行数是否一致。
7. 如果 MySQL 不可用，则通过 SQLite 兼容表结构继续导入和测试。

## 八、Spark 数据分析与查询支持

当前数据分析部分采用 Scala/Spark 作为主要计算引擎。Python 爬虫只负责采集网页/API 数据并写入 Kafka，Kafka 负责消息缓冲和传输；Spark Structured Streaming 从 Kafka 原始 Topic 中读取价格和气象事件，完成实时清洗、标准化、去重和初步统计，并将有效数据写入 MongoDB。对于历史 CSV 数据，系统通过导入脚本写入 MongoDB，后续可由 Spark 进行批量统计分析或同步到 HDFS 归档。

### 8.1 数据读取

Spark 分析的数据来源包括 Kafka 实时数据流、MongoDB 中的历史清洗结果以及本地 CSV/HDFS 归档数据。实时链路中，Spark 直接消费 `raw_price_topic` 和 `raw_weather_topic`，读取 Python 爬虫发送的 JSON 原始消息；离线分析时，可以从 MongoDB 的 `price_data`、`weather_data` 集合或历史 CSV 文件中读取数据，保证实时数据和历史数据使用统一字段口径。

### 8.2 数据处理

Spark 使用 DataFrame 和 Structured Streaming 对数据进行清洗与转换。处理内容包括字段标准化、日期格式转换、价格单位统一、缺失值过滤、异常价格过滤、重复数据去除和气象字段规范化。价格数据按 `product_name + market_name + region + date + unit` 生成确定性 `_id`，气象数据按 `region + date` 生成确定性 `_id`，避免同一业务数据重复入库。

### 8.3 数据分析

清洗后的数据支持实时统计和离线分析两类场景。实时统计可计算最新采集批次的数据量、地区覆盖数、产品覆盖数、平均价格、最高价、最低价和异常波动记录；离线分析可按日期、地区、产品和市场维度计算价格均值、中位数、标准差、环比变化率和波动区间，并结合气象数据分析降雨量、温度、湿度等因素与价格变化之间的关系。

| 分析类型 | 关键字段 | 业务用途 |
| --- | --- | --- |
| 数据总量统计 | source_topic、ingested_at、date | 展示历史数据和实时采集规模 |
| 产品价格趋势 | product_name、date、average_price | 绘制产品价格趋势折线图 |
| 地区价格对比 | region、average_price | 分析不同地区价格差异 |
| 市场价格对比 | market_name、product_name、date | 支持批发市场维度监控 |
| 气象影响分析 | region、date、temperature、rainfall、humidity | 分析气象因素与价格波动关系 |
| 异常波动识别 | average_price、change_rate、warning_level | 识别价格突增、突降和异常记录 |

### 8.4 结果存储

Spark 清洗后的明细数据写入 MongoDB，核心集合包括 `agri_price.price_data` 和 `agri_price.weather_data`。统计分析结果、预警结果和预测结果可继续写入 MongoDB 的分析类集合，供后续 Spring Boot API 查询；首页指标、热门产品价格和最新预警信息可缓存到 Redis，提高前端访问速度；原始数据和长期历史数据可保留在 CSV/HDFS 中，便于追溯和离线批处理。

## 九、存储资源需求估算

### 9.1 当前本组数据存储规模

当前清洗后 CSV 文件总大小约为：

| 数据类型 | 文件大小 |
| --- | ---: |
| 价格数据 CSV | 167.16 KB |
| 气象数据 CSV | 37.68 KB |
| 合并数据 CSV | 239.70 KB |
| 合计 | 444.54 KB |

导入 MySQL 后，按照当前 2892 条核心入库记录估算，price_data 和 weather_data 两张表连同索引的实际占用预计小于 5 MB。为了保留原始数据、清洗数据、数据库文件、备份和模型文件，本组本地环境建议预留至少 5 GB 磁盘空间。

### 9.2 实训平台扩展规模估算

若按实训平台目标进行扩展，假设采集对象包括 500 种农产品、1520 个数据来源点，每 30 分钟采集一次，则年度记录数可粗略估算为：

```text
500 种农产品 × 1520 个来源 × 48 次/天 × 365 天 ≈ 13,315,200,000 条/年
```

按每条原始记录约 1 KB 估算，年度原始数据约 13.3 TB。若 HDFS 采用 3 副本策略，原始数据存储需求约 40 TB；再考虑清洗结果、索引、日志、预测结果和备份，实训平台建议规划不低于 100 TB 的总存储容量。这个估算与实训任务书中“总存储容量不低于 100TB”的目标相匹配。

### 9.3 分层容量建议

| 存储层 | 建议容量 | 说明 |
| --- | ---: | --- |
| Kafka | 500 GB - 2 TB | 保存短期采集消息，按保留时间滚动清理 |
| HDFS | 60 TB - 80 TB | 保存原始数据、清洗数据和历史归档，建议 3 副本 |
| MySQL | 500 GB - 2 TB | 保存热点结构化数据、配置表、统计表和预测结果 |
| Redis | 16 GB - 64 GB | 缓存首页指标、热门查询和预警信息 |
| 备份空间 | 20 TB 以上 | 保存重要数据快照和数据库备份 |

## 十一、风险与优化措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 数据来源不稳定 | 爬虫无法持续获取数据 | 准备多个公开数据源，并保留 CSV 数据备份 |
| 字段格式不统一 | 入库失败或分析结果偏差 | 在清洗阶段统一字段名、日期格式和价格单位 |
| MySQL 服务不可用 | 数据无法写入主库 | 启用 SQLite 降级方案，保证本地演示继续进行 |
| 数据重复导入 | 统计结果偏大 | 入库前清空目标表，后续增加唯一约束或导入日志 |
| 数据量扩大后查询变慢 | 前端图表加载慢 | 保留 product_name + date、region + date 索引，并使用 Redis 缓存热点数据 |
| HDFS/Kafka 环境搭建复杂 | 影响实训进度 | 当前阶段使用 CSV 和 pandas 模拟数据链路，后续迁移到实训集群 |
| 数据丢失 | 影响复现实验结果 | 保留 raw、processed、数据库和测试记录多份备份 |

## 十二、结论

按照农产品价格采集监控平台的实训要求，完整数据存储方案应采用 Kafka、HDFS、Spark、MySQL、Redis 等组件形成分层架构，分别满足实时采集缓冲、历史数据归档、清洗分析、结构化查询和热点缓存需求。

结合本组当前项目规模和本地运行条件，本次实操采用 CSV 文件归档、MySQL 主数据库和 SQLite 降级数据库完成核心落地。该方案能够支撑当前农产品价格数据和气象数据的批量入库、基础查询、价格气象关联分析、模型训练和可视化展示，同时在设计上保留了向实训平台完整大数据架构扩展的路径。
