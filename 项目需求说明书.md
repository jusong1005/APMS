# 农产品价格采集监控平台项目需求说明书

## 一、项目名称

**农产品价格采集监控平台**

## 二、项目背景

随着农业产业数字化转型不断推进，农产品价格信息已经成为农户、经销商、农业企业和管理部门进行生产经营决策的重要依据。农产品价格受季节、天气、产地、市场供需、物流运输和政策调控等因素影响，波动频繁且区域差异明显。

传统价格信息获取方式主要依赖人工查询、市场公告或零散网站信息，存在数据来源分散、更新滞后、格式不统一、分析深度不足等问题。为了提高农产品价格信息的采集效率和监控能力，本项目拟建设农产品价格采集监控平台，利用爬虫/API 数据采集、数据清洗、结构化存储、统计分析、机器学习预测和可视化展示等技术，实现农产品价格数据的统一管理和监控分析。

本项目同时作为大数据综合实训项目，要求小组成员在项目过程中掌握数据采集、数据处理、数据库设计、后端接口、前端可视化、模型预测和测试验收等完整开发流程。

## 三、项目建设目标

### 3.1 实训目标

1. 掌握农产品价格数据采集方法，包括公开数据下载、API 调用和简单网页采集。
2. 掌握数据清洗处理方法，包括缺失值处理、重复数据删除、字段标准化和单位转换。
3. 掌握数据存储设计方法，能够根据业务场景设计 MongoDB 集合结构，并了解 HDFS、Kafka、Redis 等大数据组件在完整平台中的作用。
4. 掌握基础统计分析和机器学习预测方法，能够分析价格趋势、地区差异和价格波动。
5. 掌握前后端协作和可视化展示方法，能够用 Vue 页面展示价格趋势、地区对比和预测结果。
6. 掌握项目测试和文档编写方法，形成可验收的实训成果。

### 3.2 平台目标

1. 实现农产品价格数据和辅助气象数据的统一采集与管理。
2. 构建稳定的数据清洗和存储流程，保证入库数据字段完整、格式统一、记录数一致。
3. 支持按农产品、地区、日期、市场等条件查询价格数据。
4. 支持价格趋势分析、地区价格对比、气象因素影响分析和异常波动识别。
5. 支持基于历史价格和辅助特征的价格预测，为用户提供未来价格走势参考。
6. 通过前端可视化页面展示核心指标、图表和预测结果。
7. 在本地环境中提供 SQLite 降级方案，保证演示和测试流程可复现。

## 五、总体架构需求

平台采用分层架构设计，从上到下包括前端展示层、后端服务层、实时采集层、数据处理层和数据存储层。当前单节点部署以 Python 爬虫、Kafka、Scala/Spark 清洗、MongoDB、Redis、Spring Boot 预留后端、Vue 前端和 Nginx 为核心，Hadoop/HDFS 用于完整实训平台扩展。

### 5.1 技术架构分层

| 层级 | 主要内容 | 功能说明 |
| --- | --- | --- |
| 前端展示层 | Vue 3、ECharts、Axios | 展示首页指标、价格趋势、地区对比、气象影响、预警信息和预测结果 |
| 后端服务层 | Spring Boot 3.x | 提供数据查询、统计分析、预测结果、任务状态和预警接口，后续统一使用 8080 端口 |
| 实时采集层 | Python 爬虫、requests/BeautifulSoup/Playwright | 定时或实时抓取网页/API 数据，将原始价格数据转换为 JSON 消息并写入 Kafka |
| 消息缓冲层 | Kafka 3.7.0 | 接收原始采集数据，按 Topic 缓冲实时消息，解耦爬虫和后续处理消费者 |
| 数据处理层 | Scala 2.12.18、Spark 3.5.1 Structured Streaming | 从 Kafka 消费消息，完成字段标准化、缺失值处理、异常值过滤、去重、单位统一和统计分析 |
| 数据存储层 | MongoDB 7.0.12、Redis 7.2.5、CSV、HDFS 扩展 | MongoDB 保存清洗后的业务数据，Redis 缓存热点指标和预警结果，CSV/HDFS 保存归档和历史数据 |
| 部署访问层 | Nginx 1.24.0 | 托管前端静态资源，后续将 API 请求反向代理到 Spring Boot 服务 |

### 5.2 核心技术组件职责

| 组件 | 主要职责 | 当前规划 |
| --- | --- | --- |
| Python 爬虫 | 采集农产品价格网页/API 数据，完成基础字段提取 | 作为实时采集生产者，将数据写入 Kafka 原始 Topic |
| Kafka | 接收、缓冲和分发实时采集消息 | Kafka 不承担清洗逻辑，只负责消息传输和削峰 |
| Scala/Spark 清洗作业 | 消费 Kafka 原始数据并执行清洗规则 | 使用 Scala 2.12.18 + Spark Structured Streaming 实现，适合体现大数据实训链路 |
| Spark | 实时清洗、批量清洗、离线统计和大规模数据分析 | 与 Scala 2.12.18、JDK 17 配套使用 |
| MongoDB | 保存清洗后的价格、气象、分析、预警和预测结果 | 作为主数据库，使用 agri_price 库和业务集合 |
| Redis | 缓存首页指标、热门产品、最新价格和预警结果 | 提升前端高频查询响应速度 |
| Spring Boot | 提供 RESTful API、任务管理和后续 WebSocket 推送 | 后续替代当前轻量后端接口，连接 MongoDB/Redis |
| Vue/ECharts | 提供前端可视化页面 | 展示趋势图、对比图、预警列表和预测结果 |

### 5.3 实时数据链路

平台实时数据链路设计如下：

```text
Python 爬虫
  -> Kafka 原始数据 Topic
  -> Scala/Spark 数据清洗作业
  -> MongoDB 持久化
  -> Redis 热点缓存
  -> Spring Boot API
  -> Vue 前端展示
```

其中，Kafka 只负责消息缓冲和传输，不负责清洗；数据清洗发生在 Kafka 后面的 Scala/Spark 作业中。清洗后的有效数据进入 MongoDB，热点指标和预警结果进入 Redis，前端通过 Spring Boot API 获取展示数据。

## 六、功能需求

### 6.1 数据采集模块

系统需要采集农产品价格数据和辅助气象数据。

#### 6.1.1 农产品价格数据字段

| 字段名称 | 字段说明 |
| --- | --- |
| product_name | 农产品名称，如番茄、玉米、苹果等 |
| product_category | 农产品类别，如蔬菜类、水果类、粮食类 |
| market_name | 市场名称或数据来源平台 |
| region | 所属地区 |
| date | 数据日期或采集日期 |
| highest_price | 最高价格 |
| lowest_price | 最低价格 |
| average_price | 平均价格或单价 |
| unit | 价格单位，如元/公斤 |

#### 6.1.2 气象辅助数据字段

| 字段名称 | 字段说明 |
| --- | --- |
| region | 地区 |
| date | 日期 |
| average_temperature | 平均气温 |
| highest_temperature | 最高气温 |
| lowest_temperature | 最低气温 |
| rainfall | 降雨量 |
| humidity | 空气湿度 |
| sunshine_duration | 日照时长 |
| weather_condition | 天气状况 |

#### 6.1.3 采集要求

1. 至少覆盖 3 种农产品、2 个以上地区和 6 个月以上时间范围。
2. 价格数据记录数不少于 1000 条。
3. 数据来源需要记录来源名称、字段说明和时间范围。
4. 采集数据应保存为 CSV 文件，便于检查和复现实验。
5. 完整平台可扩展为 Python 定时或实时爬虫采集，并通过 Kafka 写入实时数据流。
6. 实时爬虫应将采集结果封装为 JSON 消息，至少包含 product_name、product_category、market_name、region、collect_time、price、unit、source_url 等字段。
7. 爬虫模块应记录数据源、采集时间、请求状态和失败原因，便于后续任务监控和问题排查。

### 6.2 数据处理模块

系统应具备以下数据处理能力。数据清洗模块采用 Scala/Spark Structured Streaming 从 Kafka 中消费 Python 爬虫采集的原始农产品价格数据，对数据进行字段标准化、缺失值处理、异常值过滤、重复数据去除和单位统一等操作。清洗后的有效数据写入 MongoDB，供后端接口查询和前端展示使用；对于历史统计、批量分析和模型训练数据，可进一步写入 HDFS 或由 Spark 进行离线分析处理。

1. 对缺失值进行识别和处理。
2. 删除重复数据，避免重复入库导致统计偏差。
3. 统一字段名称、日期格式和价格单位。
4. 检查价格字段是否为有效数值。
5. 按日期、地区和农产品名称对价格数据进行整理。
6. 将气象数据与价格数据按 region 和 date 关联。
7. 生成清洗后的 processed_price.csv、processed_weather.csv 和 merged_data.csv。
8. 从 Kafka 原始数据 Topic 消费实时数据，输出清洗后的标准化 JSON 文档。
9. 将不合法数据写入异常数据 Topic 或异常记录集合，便于质量追踪。
10. 对清洗过程记录处理数量、成功数量、失败数量和处理时间。

### 6.3 数据存储模块

数据存储模块需要完成以下功能：

1. 使用 MongoDB 保存清洗后的农产品价格数据。
2. 使用 MongoDB 保存气象辅助数据、分析结果、预警结果和预测结果。
3. 保留 CSV 文件作为原始数据和清洗结果归档。
4. 在 MongoDB 不可用时保留 CSV 或 SQLite 作为本地测试降级方案。
5. 设计 product_name + date、region + date、market_name + collect_time 等索引，提高查询效率。
6. 完整平台可扩展 HDFS 保存历史数据，Redis 缓存热点指标，Kafka 支撑实时采集。
7. 推荐 MongoDB 数据库名称为 agri_price，核心集合包括 price_data、weather_data、prediction_result、quality_alert、collector_task。

### 6.4 数据查询模块

系统应支持以下查询功能：

1. 按农产品名称查询历史价格数据。
2. 按地区查询价格数据。
3. 按日期范围查询价格走势。
4. 按市场名称查询价格数据。
5. 查询指定地区的气象数据。
6. 查询价格数据与气象数据的关联结果。
7. 查询某一农产品在不同地区的平均价格和价格波动情况。

### 6.5 数据统计分析模块

系统需要完成以下分析任务：

1. 统计农产品平均价格、最高价格和最低价格。
2. 分析指定农产品在不同日期的价格变化趋势。
3. 对比不同地区同一农产品的价格水平。
4. 分析价格在不同月份或季节中的变化规律。
5. 分析温度、降雨量、湿度等气象因素与价格之间的关系。
6. 识别价格上涨、下跌和异常波动情况。
7. 当价格波动超过设定阈值时，可作为后续预警功能的数据基础。

### 6.6 价格预测模块

系统应根据历史数据建立预测模型，实现农产品未来价格预测。主要功能包括：

1. 选择预测农产品。
2. 选择预测地区。
3. 选择预测时间范围。
4. 使用历史价格和气象特征训练模型。
5. 输出预测价格结果。
6. 展示真实价格与预测价格对比。
7. 计算 MAE、MSE、RMSE 等误差指标。

可选模型包括线性回归、随机森林回归、ARIMA 时间序列模型和 Spark MLlib 回归模型。本组优先采用随机森林回归和 ARIMA，兼顾实现难度和展示效果。

### 6.7 可视化展示模块

平台前端需要通过图表展示分析结果和预测结果，主要功能包括：

1. 首页展示数据总量、农产品数量、地区数量和平均价格等核心指标。
2. 使用折线图展示农产品价格趋势。
3. 使用柱状图展示不同地区价格对比。
4. 使用热力图或相关性图展示气象因素与价格关系。
5. 使用折线图展示真实价格与预测价格对比。
6. 支持按产品、地区和时间范围筛选数据。
7. 完整平台可增加爬虫任务状态、价格预警和热门产品排行展示。

### 6.8 用户与权限模块扩展

完整实训平台可增加用户管理功能：

1. 管理员可管理爬虫任务、数据源和系统配置。
2. 普通用户可查询价格数据和查看可视化图表。
3. 数据分析人员可查看清洗结果、统计分析和预测结果。
4. 系统可记录用户操作日志，便于审计和问题追踪。

本组当前重点完成数据处理、存储、分析、预测和展示，用户权限作为后续扩展功能。

## 七、非功能需求

### 7.1 性能需求

1. 常规数据查询和统计分析应在可接受时间内完成。
2. 前端页面加载应保持流畅，图表展示不应明显卡顿。
3. MongoDB 集合应建立合理索引，支持按产品、地区、市场和日期筛选。
4. 完整平台中高频查询可使用 Redis 缓存，实时采集可使用 Kafka 解耦。

### 7.2 可靠性需求

1. 数据处理过程应保留原始数据和清洗后数据，便于回溯。
2. 入库后应验证数据库记录数与输入数据行数是否一致。
3. MongoDB 不可用时应保留 CSV 或 SQLite 降级验证方案。
4. 重要数据文件和报告应进行备份。

### 7.3 安全性需求

1. 数据文件应统一管理，避免误删或篡改。
2. 数据库连接信息不应硬编码到公开文档中。
3. 若后续加入用户登录，应对密码进行加密存储。
4. 管理端功能应限制权限，避免非授权用户修改爬虫任务或删除数据。

### 7.4 可扩展性需求

1. 支持增加更多农产品种类。
2. 支持增加更多地区和市场数据。
3. 支持接入更多数据源，如产量、交易量、物流和节假日数据。
4. 支持扩展 Kafka、HDFS、Spark、Redis、MongoDB 等完整大数据平台组件。
5. 支持替换或增加新的预测模型。

### 7.5 可用性需求

1. 页面操作流程应清晰，查询条件应易于理解。
2. 图表标题、坐标轴和数值单位应明确。
3. 报告和测试记录应能够说明数据来源、处理过程和测试结论。
4. 本地演示应提供明确的启动和测试命令。

## 八、数据流程设计

系统总体流程如下：

```text
Python 爬虫采集网页/API 数据
  ↓
Kafka 原始数据 Topic
  ↓
数据清洗消费者进行字段标准化、去重、缺失值处理和异常过滤
  ↓
MongoDB 保存清洗后的价格、气象、分析和预测数据
  ↓
Redis 缓存热门价格、首页指标和预警结果
  ↓
Spring Boot 提供查询、统计、预警和预测接口
  ↓
Vue/ECharts 前端可视化展示
  ↓
测试验收与报告提交
```

完整平台可扩展为：

```text
Python 爬虫 -> Kafka -> Scala/Spark 清洗 -> MongoDB/Redis -> Spring Boot API -> Vue 可视化
```

对于课程演示场景，Python 只负责网页/API 采集并发送 Kafka；Kafka 后续的数据清洗、异常分流和写入 MongoDB 由 Scala/Spark 作业完成。当数据量扩大或需要体现完整大数据链路时，可继续使用 Spark 读取 Kafka 或 HDFS 数据进行批量统计和离线分析。
