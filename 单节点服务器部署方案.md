# 农产品价格采集监控平台单节点服务器部署方案

## 一、是否可以单节点部署

可以。对于本项目的课程验收、演示和小规模数据处理，单节点服务器部署是更稳妥的方案。当前部署口径已将主数据库切换为 MongoDB，不启动 Flask 服务；Python 只负责网页/API 采集并写入 Kafka，Kafka 后续清洗和入库由 Scala/Spark Structured Streaming 完成，后端接口后续改为 Spring Boot。Hadoop、Spark、Kafka、Redis 属于完整实训平台组件，可以在同一台服务器上按伪分布式或单实例方式部署。

单节点部署适合以下场景：

1. 只有一台云服务器或实验室服务器。
2. 主要目标是完成项目演示、接口访问、数据入库和前端展示。
3. 需要在报告中体现 Hadoop、Spark、Kafka、Redis 的完整链路，但不追求生产级高可用。
4. 数据规模较小，当前 processed 数据和 MongoDB 集合足以支撑分析与可视化。

单节点部署不适合以下场景：

1. 需要 NameNode、ResourceManager、Kafka、MongoDB、Redis 的生产级高可用。
2. 需要承载长期高并发访问。
3. 需要处理 TB 级以上历史数据。

## 二、推荐服务器配置

| 项目 | 最低配置 | 推荐配置 | 说明 |
| --- | ---: | ---: | --- |
| CPU | 4 核 | 8 核及以上 | 同时运行 MongoDB、后端、前端和可选大数据组件 |
| 内存 | 16 GB | 32 GB 及以上 | Hadoop/Spark/Kafka 同机部署时建议 32 GB |
| 磁盘 | 100 GB | 300 GB 及以上 SSD | 保存依赖、日志、MongoDB、HDFS、Kafka 数据 |
| 操作系统 | Ubuntu 22.04 LTS / Rocky Linux 8 | Ubuntu 22.04 LTS | Linux 更适合大数据组件部署 |
| 网络 | 公网或校园网可访问 | 固定 IP | 便于前端和后端演示访问 |

如果服务器只有 2 核 8 GB，也可以只部署项目轻量版：MongoDB、Python 采集脚本、Vue 静态页面和 Nginx，不建议同时启动 Hadoop、Spark 和 Kafka。

本仓库根目录已提供自动化脚本 /home/ubuntu/agri/install-single-node.sh，用于完成 JDK、Maven、Scala、Node.js、Hadoop、Spark、Kafka、Redis、MongoDB 和 mongosh 的下载、解压和基础配置。脚本执行完成后，会生成 /home/ubuntu/agri/servers/env.sh，后续命令执行前先加载该环境变量文件。

## 六、推荐部署顺序

### 6.1 第一阶段：先跑通项目轻量版

第一阶段只部署当前代码真正需要的组件，优先保证项目能采集、能进入 Kafka、能清洗入库、能展示。

1. 将本阶段需要的 JDK、Maven、Node.js 等压缩包统一下载到 /home/ubuntu/agri/software；如果准备一次性离线部署，也可以同时下载 Hadoop、Spark、Kafka、Redis 压缩包。
2. 将压缩包统一解压到 /home/ubuntu/agri/servers，并配置 JAVA_HOME、MAVEN_HOME、NODE_HOME 等环境变量。
3. 安装系统依赖：Python 3.10/3.11、Nginx；MongoDB 通过压缩包部署。
4. 验证 Java 与 Maven 环境，确保 java -version 显示 17，mvn -v 正常输出。
5. 创建 MongoDB 数据库 agri_price，并按 price_data、weather_data、invalid_events 等集合设计清洗结果；backend/schema.sql 不再直接适用。
6. 在项目根目录创建 Python 虚拟环境，安装 requirements.txt。
7. 配置环境变量：KAFKA_BOOTSTRAP_SERVERS、KAFKA_RAW_PRICE_TOPIC、KAFKA_RAW_WEATHER_TOPIC。
8. 使用 Python 生产者将网页/API 原始数据写入 Kafka，再用 Scala/Spark 清洗作业写入 MongoDB。
9. 在 frontend 目录安装依赖并构建前端静态文件。
10. 使用 Nginx 托管前端 dist；当前不配置 Flask 反向代理，后续 Spring Boot 改造完成后再将 API 反向代理到 8080。

完成第一阶段后，平台已经可以支撑课程演示和答辩。

### 6.2 第二阶段：预留 Spring Boot 后端环境

本阶段只准备环境，不改代码、不创建新的 Java 后端工程。Spring Boot 不需要像 MongoDB、Redis 那样作为系统服务单独安装，通常是在后续 Java 项目的 pom.xml 中通过 spring-boot-starter-web、spring-boot-starter-data-mongodb 等依赖引入。

建议预留以下约定：

1. JDK 17 作为默认 Java 环境。
2. Maven 3.9.x 作为 Java 项目构建工具。
3. 后续 Spring Boot 项目默认端口使用 8080。
4. Spring Boot 数据库连接使用 agri_price 库。
5. Nginx 当前只托管前端静态文件，后续 Spring Boot 改造完成后再反向代理到 8080。

这样可以先完成服务器环境准备，同时不影响当前 Python 采集、Scala/Spark 清洗和前端静态页面演示。

### 6.3 第三阶段：按需增加 Redis

Redis 是可选组件。当前数据规模较小时，后端直接查 MongoDB 也可以满足展示需求。如果需要体现缓存层，可将以下数据放入 Redis：

1. 首页统计指标。
2. 热门农产品最近价格。
3. 价格预警结果。
4. 高频查询结果。

单节点部署时 Redis 使用默认单实例即可，不需要 Sentinel 或 Cluster。

### 6.4 第四阶段：按需增加 Hadoop/Spark/Kafka

如果课程要求展示完整大数据平台链路，可以在同一台服务器上部署伪分布式组件：

1. Hadoop 使用 3.4.1 单节点伪分布式，HDFS replication 设置为 1。
2. Spark 使用 spark-3.5.1-bin-hadoop3，可运行 local 模式或提交到本机 YARN。
3. Kafka 使用 kafka_2.13-3.7.0 单 broker，ZooKeeper 模式，内部 topic 副本数设置为 1。
4. 创建演示 Topic：raw_price_topic、raw_weather_topic。
5. 创建 HDFS 目录：/agri_price/raw、/agri_price/processed、/agri_price/merged。

当前项目已补充 Python 实时采集生产者和 Scala/Spark 清洗模块。Python 将网页/API 采集结果写入 Kafka 原始 Topic，Scala/Spark Structured Streaming 从 Kafka 消费消息，完成字段校验、标准化和异常分流，最后写入 MongoDB。HDFS 可作为历史归档扩展。

## 九、推荐结论

如果只有一台服务器，建议采用“两步走”：

1. 先部署项目轻量版：MongoDB、Python 采集环境、Scala/Spark 清洗环境、Vue 前端、Nginx，并同步安装 JDK 17 和 Maven，为后续 Spring Boot 改造预留环境；当前不启动 Flask 服务。
2. 后续改造后端时，再新建 Spring Boot 3.x 工程，端口建议使用 8080，数据库连接 agri_price。
3. 再按展示需要增加 Redis 7.2.5、Hadoop 3.4.1、Spark 3.5.1、Kafka 3.7.0 单节点伪分布式环境。这一部分用于体现完整大数据平台架构和后续扩展能力，并且更适合统一 JDK 17 口径。

这样既能保证平台可运行、可展示、可验收，又能避免一开始被 Hadoop/Spark/Kafka 的环境问题拖慢进度。
