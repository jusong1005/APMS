# 数科23-13第一组技术选型分析报告

## 一、项目基本信息

| 项目 | 内容 |
| --- | --- |
| 项目名称 | 农产品价格采集监控平台 |
| 小组 | 数科23-13第一组 |
| 报告类型 | 技术选型分析报告 |
| 项目目标 | 完成农产品价格数据采集、清洗、存储、分析、预测和可视化展示，形成符合实训要求的监控平台原型 |

## 二、实训技术需求概述

农产品价格采集监控平台需要围绕农产品价格信息完成从采集到展示的完整数据链路。根据实训说明书和本组当前实现，系统主要涉及以下技术能力：

1. 多源数据采集：从公开网站、批发市场页面、CSV 文件或 API 获取农产品价格和辅助气象数据。
2. 数据清洗处理：处理缺失值、异常值、重复值，统一日期格式、地区字段和价格单位。
3. 实时与批量存储：当前链路采用 Kafka 接收实时消息，MongoDB 保存清洗后的业务数据，Redis 缓存热点指标，CSV/HDFS 用于历史归档和离线扩展。
4. 数据统计分析：完成价格趋势、地区差异、气象影响和价格波动分析。
5. 机器学习预测：基于历史价格和辅助特征预测未来价格走势。
6. Web 可视化展示：通过 Vue 前端和图表组件展示核心指标、价格趋势和预测结果。
7. 测试与验收：完成数据完整性、格式正确性、入库一致性、基础查询和可视化展示测试。

## 三、技术选型原则

本项目属于大数据综合实训项目，技术选型需要同时考虑实训要求和本组实际落地能力。

1. 满足实训要求：技术路线应覆盖采集、清洗、存储、分析、预测和展示全流程。
2. 保证可运行：核心功能优先采用本地可运行、可测试的方案实现。
3. 体现大数据思想：报告和架构中保留 Kafka、HDFS、Spark、Redis 等完整平台扩展设计。
4. 控制实现难度：避免在课程周期内强行搭建过重环境导致核心功能无法完成。
5. 便于验收展示：优先选择能生成图表、接口、测试结果和文档成果的技术。
6. 具备扩展能力：后续可接入更多农产品、地区、市场和预测模型。

## 五、分模块技术选型分析

### 5.1 数据采集技术选型

| 备选技术 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 公开 CSV 数据 | 获取简单、格式稳定、便于复现 | 实时性较弱 | 课程项目初始数据集 |
| requests + BeautifulSoup | 轻量灵活，适合采集简单网页 | 需要处理网页结构变化 | 批发市场价格页面采集 |
| Scrapy | 爬虫框架完整，适合任务管理和扩展 | 学习和配置成本较高 | 完整实训平台的多源爬虫 |
| API 调用 | 数据结构清晰，便于定时采集 | 可能需要 Key 或额度限制 | 天气和公开数据接口 |
| Kafka 接入 | 适合实时数据流和高吞吐采集 | 环境部署复杂 | 完整平台实时采集扩展 |

**最终选型：公开 CSV/API + requests 作为当前实现，Scrapy + Kafka 作为完整平台扩展。**

选型依据：本组当前需要保证数据集可获取、可清洗、可入库，因此以公开 CSV 和 API 数据为主；若后续继续完善采集监控平台，可使用 Scrapy 管理爬虫任务，并将采集结果写入 Kafka。

### 5.2 数据清洗处理技术选型

| 备选技术 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| pandas | 上手快，适合历史 CSV 导入和小规模校验 | 不承担实时主链路清洗 | 历史数据导入辅助 |
| NumPy | 数值计算效率高 | 表格清洗能力弱于 pandas | 统计计算和特征处理辅助 |
| Scala + Spark Structured Streaming | 适合实时消费 Kafka 并进行流式清洗 | 需要配置 JDK、Scala、Spark 环境 | 当前实时清洗和分析主链路 |
| Spark SQL | SQL 风格分析清晰，可处理批量和流式数据 | 依赖 Spark 运行环境 | 离线统计分析和批量汇总 |

**最终选型：Scala + Spark Structured Streaming 为主，pandas 仅用于历史 CSV 导入和辅助校验。**

当前项目已搭建 Kafka、Scala 和 Spark 单节点环境，实时数据由 Python 爬虫写入 Kafka，再由 Spark 消费原始 Topic 完成字段统一、缺失值处理、重复值删除、异常过滤和 MongoDB 入库。历史 CSV 数据通过导入脚本写入 MongoDB，后续也可以由 Spark 读取 MongoDB 或 HDFS 数据完成批量分析。

### 5.3 数据存储技术选型

| 备选技术 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| CSV 文件 | 简单直观，便于归档和检查 | 不适合复杂查询和并发访问 | 原始数据和清洗结果备份 |
| MongoDB | 文档结构灵活，适合保存价格、气象、预警和预测结果 | 需要设计索引和去重键 | 当前主数据库 |
| SQLite | 无需服务端，便于测试和演示 | 不作为当前主链路组件 | 本地降级和简单验证 |
| HDFS | 适合海量历史数据归档 | 部署维护成本高 | 完整实训平台历史数据层 |
| Redis | 查询快，适合缓存热点数据 | 不适合作为唯一持久化存储 | 首页指标、热门价格缓存 |
| Kafka | 高吞吐消息队列 | 不是长期数据库 | 实时采集数据缓冲 |

**最终选型：MongoDB + Redis + CSV/HDFS 为当前存储方案，Kafka 作为实时消息缓冲组件。**

### 5.4 后端服务技术选型

| 备选技术 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| Flask | 轻量、开发快 | 已从当前后端方案中移除 | 早期原型参考 |
| FastAPI | 性能好，自动生成接口文档 | 需要学习异步和类型声明 | 后续接口扩展 |
| SpringBoot | 企业级成熟，贴合实训平台文档 | Java 工程量较大 | 完整实训标准平台 |

**最终选型：Spring Boot 3.x 作为后续后端服务实现。**

当前已删除旧 Flask 后端，后续后端统一使用 Spring Boot 连接 MongoDB 和 Redis，为 Vue 前端提供数据查询、任务管理、统计分析和预警接口。

### 5.5 前端可视化技术选型

| 备选技术 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| Vue 3 + Vite | 工程化清晰，交互能力强 | 需要前端开发基础 | 当前平台前端实现 |
| ECharts | 图表类型丰富，适合数据看板 | 需要配置图表参数 | 趋势图、柱状图、相关性图 |
| DataV | 大屏组件丰富 | 样式定制需适配 | 完整监控大屏扩展 |
| Matplotlib | 适合生成静态报告图 | 交互性弱 | 报告图表输出 |
| Plotly | 交互图表较好 | 与当前 Vue 工程衔接需额外处理 | 数据探索和独立图表 |

**最终选型：Vue 3 + ECharts 为主，DataV 作为监控大屏扩展，Matplotlib/Plotly 用于报告图表辅助。**

### 5.6 价格预测模型选型

| 备选模型 | 优点 | 缺点 | 适用场景 |
| --- | --- | --- | --- |
| 线性回归 | 实现简单，解释性强 | 拟合复杂非线性能力有限 | 基线模型 |
| 随机森林回归 | 能处理非线性关系，对特征尺度不敏感 | 解释性弱于线性模型 | 当前主模型 |
| ARIMA | 适合时间序列趋势预测 | 对数据平稳性有要求 | 单产品历史价格预测 |
| LSTM | 适合复杂序列建模 | 数据量和训练成本较高 | 后续深度学习扩展 |
| Spark MLlib | 适合大规模建模 | 依赖 Spark 环境 | 完整大数据平台扩展 |

**最终选型：随机森林回归为主，ARIMA 和线性回归作为对比模型，Spark MLlib 作为扩展。**

## 八、技术风险与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 爬虫数据源变化 | 采集失败或字段缺失 | 保留公开 CSV 数据作为备份，采集脚本增加字段校验 |
| 大数据环境搭建复杂 | 影响实训进度 | 当前先完成本地可运行版本，HDFS/Kafka/Spark 作为扩展设计 |
| 数据字段不统一 | 入库失败或统计结果错误 | 在清洗阶段统一字段名、日期格式和价格单位 |
| MongoDB 连接失败 | 数据无法写入主库 | 保留 CSV 历史数据并检查 MongoDB 服务状态 |
| 模型预测效果一般 | 预测误差较大 | 增加时间、地区、气象特征，并用多模型对比 |
| 前端展示时间不足 | 影响答辩效果 | 优先完成核心图表：趋势、地区对比、气象影响、预测对比 |
| 文档与代码口径不一致 | 验收时解释困难 | 文档统一采用“实训标准架构 + 本组落地实现”的表达 |

## 九、结论

本项目最终采用“Python 爬虫 + Kafka + Scala/Spark 清洗分析 + MongoDB/Redis 存储 + Spring Boot 预留后端 + Vue/ECharts 可视化”的单节点可运行技术路线，并保留 HDFS 和 Spark SQL 离线分析作为后续扩展方案。

该技术路线既能够满足农产品价格采集监控平台实训对采集、清洗、存储、分析、预测和展示的要求，又能保证本组在课程周期内完成可运行、可测试、可展示的项目成果。
